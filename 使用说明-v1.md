# Paper Novelty Design V1 使用说明

## 1. Skill 是做什么的

`paper-novelty-design-v1` 用于把一个还不够成熟的论文创新点，整理成更正式、更抗审稿质疑的结构化贡献表达。

它主要解决以下几类问题：

1. 创新点说不清，只有一个模糊想法。
2. 只有方法，没有清晰的研究问题和洞察。
3. 担心创新点像“工程堆料”或“换模块提点”。
4. 需要把创新点改写成论文引言里的 contribution。
5. 需要给创新点设计可证伪命题、消融实验和审稿人压力测试。
6. 需要记录创新点在实验过程中的迭代版本。

一句话总结：这个 Skill 不是帮你“润色一句话”，而是帮你把“创新点”变成“可发表叙事”。

## 3. 怎么使用

### 3.1 直接调用

如果你的 Codex / 技能系统支持显式技能调用，可以直接这样写：

```text
Use $paper-novelty-design-v1 to rewrite my idea into Problem-Method-Insight and pressure-test its novelty.
```

如果你希望中文方式使用，可以直接这样描述任务：

```text
使用 $paper-novelty-design-v1，帮我把下面的论文想法整理成 Problem-Method-Insight，
并检查它是否属于伪创新、是否可证伪、是否能写成正式 contribution。
```

### 3.2 推荐输入方式

最推荐你按三层输入：

```markdown
- 研究领域：时间序列预测
- 目标场景：长时序预测
- Problem: ...
- Method: ...
- Insight: ...
- 已有实验或观察: ...
- 最接近的相关工作: ...
- 当前担心的问题: ...
```

如果你现在还没有三层结构，也可以只给一个草稿：

```markdown
我的想法是：
...

我希望你帮我：
1. 提炼创新点
2. 判断是否像工程堆料
3. 设计 contribution
4. 设计可证伪实验
```

### 3.3 支持的输入材料

这个 Skill 适合处理：

1. 论文创新点草稿
2. 开题报告中的研究内容
3. Introduction 草稿
4. Abstract 草稿
5. 论文贡献声明初稿
6. 某个方法设计说明
7. 实验后准备重写创新点的记录

## 4. Skill 会输出什么

通常会输出以下几类结果中的一部分或全部：

1. `Problem-Method-Insight` 三层重写版本
2. 创新点是否存在伪创新风险的判断
3. 1 到 3 条可证伪命题
4. 与命题配套的消融实验或验证方案
5. 审稿人可能攻击的问题清单
6. 正式的 contribution statement
7. 一句话版、三句话版、一段话版创新点表述
8. `v1 / v2 / v3` 形式的创新点演化日志模板

## 5. 最常见的 4 种用法

### 用法 1：把一个模糊想法整理成正式创新点

示例：

```text
使用 $paper-novelty-design-v1，帮我把下面这个时序预测想法整理成正式创新点，
输出 Problem、Method、Insight 三层，并指出逻辑链是否完整。
```

### 用法 2：检查是不是伪创新

示例：

```text
使用 $paper-novelty-design-v1，判断下面这个方法更像真实创新、
工程组合，还是跨领域移植，并给出修改建议。
```

### 用法 3：把创新点改写成论文贡献声明

示例：

```text
使用 $paper-novelty-design-v1，把下面内容改写成论文引言末尾的 contributions，
要求 3 到 4 条，风格正式、具体、不夸张。
```

### 用法 4：实验做完后重构创新点

示例：

```text
使用 $paper-novelty-design-v1，根据下面实验结果，
判断原来的 Insight 是否仍然成立；如果不成立，帮我重写一个更稳妥的新版本。
```

## 6. 推荐工作流

建议按下面顺序使用：

1. 先给草稿或三层结构。
2. 先做 `Problem-Method-Insight` 重写。
3. 再做伪创新检查。
4. 再做可证伪命题和实验设计。
5. 最后再写 contributions 和摘要表述。

这样做的原因是：先把“研究到底在讲什么”理顺，再去写正式文本，质量会明显更稳。

## 7. 使用时的注意事项

1. 不要把“指标更高”直接当成 `Insight`。
2. 不要把“我是第一个”当成核心贡献。
3. 如果相关工作检索还没做完，不要过早下“真正未被做过”的结论。
4. 如果实验不支持原始假设，应该缩小 claim，而不是硬改故事去迎合结果。
5. 这个 Skill 可以帮助你形成更强的创新点叙事，但不能替代真实文献检索和真实实验。

## 8. 什么时候应该优先使用这个 Skill

当你出现下面任一情况时，优先调用它：

1. 你能描述方法，但说不清为什么做这个方法。
2. 你能写出结果，但写不出论文贡献。
3. 你感觉自己的创新点可能只是“换个模块”。
4. 你准备开题、中期、投稿，需要一版更正式的创新点表达。
5. 你已经做了实验，但发现原有叙事不够稳。

## 9. 一个完整输入示例

```markdown
使用 $paper-novelty-design-v1，帮我分析下面这个创新点。

- 研究领域：时间序列预测
- 目标任务：长时序预测
- Problem: 现有方法对强周期信号建模效率不足，长预测窗口下性能下降明显。
- Method: 我想加入一个可学习的频域门控分支，与时域主干并行建模，再自适应融合。
- Insight: 我猜测频域信息的收益依赖数据的周期强度，周期越强，频域分支越有效。
- 已有证据: 在 ETT 和 Traffic 上有提升，在 Exchange 上提升很弱。
- 我希望你输出:
  1. 三层结构修订版
  2. 是否存在伪创新风险
  3. 2 条可证伪命题
  4. 对应消融实验
  5. contribution statement
```

## 10. 一个完整输出目标示例

理想情况下，这个 Skill 输出后，你应该能直接拿去用于：

1. 开题报告里的“研究创新点”
2. Introduction 末尾的 contributions
3. 实验设计里的 ablation 计划
4. 和导师讨论时的创新点版本迭代
5. 投稿前的自检清单

一句话总结：最好的用法不是“让它替你想创新”，而是“让它把你的创新点打磨成更经得起质疑的研究叙事”。
